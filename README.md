# モジュール厳選シミュレーター (v0.6)

本プログラムは、指定された条件に基づくモジュール厳選の期待値を算出するためのシミュレーターです。

## v0.6 更新履歴
* **処理アルゴリズムの最適化**: 内部設計の抜本的な見直しを行い、Rustの演算能力の限界を引き出すアーキテクチャを採用しました。
* **パフォーマンスの向上**: v0.5と比較して、処理速度が7〜10倍に向上しています。

---

## パフォーマンスと制約
* 一般的な試行回数（ドロップ数が10,000回程度）で条件を満たせる設定であれば、1,000回のシミュレーションは通常2,30秒以内には完了します。
* 処理が一向に進まない場合は設定した条件が天文学的な確率を要求しており、現実的な厳選の範囲を逸脱していることを意味します。その場合はプロセスを停止し、条件を緩和してください。
* **統計的精度の低下**: 本シミュレーターの期待値は「成功した試行」から算出されます。上限（`max_drops_limit`）に到達した失敗試行の割合が増加するほど、算出される期待値の正確性は低下します。

## 設定ファイル (`config.json`)

シミュレーションの挙動とモジュールのキープ（選別）基準は、`config.json` にて定義されます。

### 基本設定
* `default_simulations`: デフォルトのシミュレーション実行回数（推奨: `1000`）。
* `max_drops_limit`: 1回のシミュレーションにおけるモジュール厳選の打ち切り回数。非現実的なループを防ぐための上限値であり、`10000` 程度を現実的な範囲として推奨します（最大50000までを推奨）。

### 閾値設定 (フィルタリングロジック)
* `min_extreme_sum`: モジュールに「極系効果」が含まれていた場合、その極系の値の合計に対する最低要求ライン。
* `min_total_with_extreme`: 「極系効果を含む」モジュールにおける、目標ステータス（極＋通常）の合計値の最低要求ライン。
* `min_total_without_extreme`: 「極系効果を含まない（通常ステータスのみ）」モジュールにおける、目標ステータス合計の最低要求ライン。
  * ※4枠の極々モードなど、極なしモジュールを完全に破棄したい場合は `255` に設定してください。

### 拡張条件設定
* `multi_type_trigger_candidates`: 複数種類の目標効果が含まれる場合の拡張判定を有効にするトリガー。基本は `3` を指定し、この機能を使用しない場合は `255` を指定して無効化してください。
* `min_total_for_2_types` / `min_total_for_3_types`: モジュール内に目標効果が「2種類」または「3種類」含まれている場合に適用される、個別の合計値閾値。
* `rescue_normal_only_2_types` / `rescue_normal_only_3_types`: 極モードにおいて、あえて極なしモジュールをキープするための「通常用救済閾値」。不良モジュールのプール混入を防ぐため、厳しめに設定することを推奨します。

## インベントリデータ (`inventory.csv`) の作成手順

現在所持しているモジュールをシミュレーションのプールに組み込む場合、以下の手順でデータを抽出し、所定のフォーマットで配置する必要があります。

1. [STModuleManager](https://github.com/honononta-2/STModuleManager) をダウンロードします。
2. ゲームクライアント（スタレゾ）と STModuleManager を同時に起動します。
3. STModuleManager 側の言語設定を **「英語」** に変更します。
4. ツールからモジュールのリストをCSV形式で出力します。
5. 出力したCSVファイルをスプレッドシート（Google Sheets等）にインポートします。
6. 以下の列を削除します：`Type`, `Rarity`, `Total`, `Obtained At`
7. スプレッドシートの1行目（ヘッダー行）を削除します。
8. 編集したデータをCSV形式でダウンロードします。
9. 本シミュレーターの `inventory.csv` を開き、**元の1行目（ヘッダー）はそのまま残し**、2行目以降にダウンロードしたデータをペーストして保存します。
