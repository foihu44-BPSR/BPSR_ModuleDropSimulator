設計の抜本的な見直しを図ることでRustでの限界を引き出した。
処理速度は7-10倍程度に向上。


"default_simulations": 1000,
デフォルトのシュミレーション回数
  
"max_drops_limit": 50000,
モジュールの厳選回数、10000ぐらいが現実的な範囲

期待値は成功したものから算出されるので失敗回数が増えると不正確になります。


inventory.csvの作り方

1.https://github.com/honononta-2/STModuleManagerをダウンロード

2.スタレゾと1のプログラムを同時に起動

3.プログラム側の言語設定を英語にする

4.csv形式でモジュールのリストを作成

5.スプレッドシートにインポート

6.Type,Rarity,Total,Obtained Atの列を削除

7.1行目を削除

8.csv形式でダウンロードし、inventory.csvの1行目をそのままにコピペ

スタレゾのパケットについてはまだ無知なので暇があったら自動で取得する機能を追加します。

閾値関連
min_extreme_sum(極値の最低ライン)
モジュールに極系効果が含まれていた場合、その「極系の値」が最低いくつあればキープするか

min_total_with_extreme(極を含む場合の合計最低ライン)
極系を含むモジュール全体の「目標ステータスの合計値(極＋通常)」が最低いくつあればキープするか

min_total_without_extreme(極を含まない場合の通常合計最低ライン)
極系が含まれていない(通常ステータスのみの)モジュールで目標の通常ステータスの合計が最低いくつあればキープするか。4枠の極々では255に設定する。



multi_type_trigger_candidates(より詳細な条件設定)
基本3でOK、使わない場合は255

min_total_for_2_types/min_total_for_3_types(種類数による閾値)
目標効果がモジュール内に「2種類」含まれている場合、または「3種類」含まれている場合に適用される特別な合計値の閾値。

rescue_normal_only_2_types / rescue_normal_only_3_types(極モードでの通常用の閾値)
極モードでの極無しモジュール回収閾値、厳しめに設定した方が良い。

一般的な試行回数(10000回以下)で条件を達成する場合、1000回のシュミレーションでも10分以内には終わるはずです。
一向に進まない場合はマトモな回数ではないという事なので停止させた方が良いでしょう。
